Detailed usage manual for the Mnemo CLI and memory runtime

Abstract

Mnemo is a repository-native memory system for AI coding agents. Its primary entrypoint is a Node.js CLI that scaffolds a canonical .mnemo/ memory layer, installs compatibility bridges for .cursor/ and .agent/, and optionally enables semantic retrieval with Gemini or OpenAI embeddings. This guide explains how to use Mnemo as an operator rather than as a paper reader: how to install it, how to seed memory, how to author lessons and journals, how to search memory efficiently, how vector mode works, how autonomous mode changes repository behavior, and how to keep the system healthy over time.

Keywords: Mnemo, CLI operations, memory authoring, retrieval, vector search, autonomous runtime, repo-native documentation, AI coding agents

1. System Overview

Mnemo should be understood as an operating protocol, not just an installer. The CLI creates a memory filesystem, installs rule files for agent-facing environments, drops helper scripts into scripts/memory/, and optionally enables a Python semantic-retrieval layer. The canonical memory root is .mnemo/memory/. Compatibility bridges then expose the same content under .cursor/ and .agent/ so multiple tools can share one memory model.

The system is intentionally stratified. The always-read layer is small and stable; indexed lessons capture reusable failure prevention; journals preserve chronology; digests compress history for cheap retrieval; and vector mode adds semantic recall without changing the file contract. Mnemo is therefore a retrieval order plus a file layout plus a maintenance loop.

2. Installation Protocol

The supported front door is the package CLI:

npx @dinasor/mnemo-cli@latest

That command launches an interactive wizard unless --yes is supplied or stdin is not a TTY. The wizard asks whether vector mode should be enabled, which embedding provider should be used, and whether API keys should be pulled from the environment or the project .env.

Mode When to use it Representative command
Interactive standard install First install in a repo; operator wants guided prompts. npx @dinasor/mnemo-cli@latest
Non-interactive standard install CI, scripts, or repeated repo bootstrap. npx @dinasor/mnemo-cli@latest --yes
Vector install Semantic search, MCP tools, or autonomous runtime required. npx @dinasor/mnemo-cli@latest --enable-vector --vector-provider gemini --yes
Preview only Need to inspect planned file writes before mutation. npx @dinasor/mnemo-cli@latest --dry-run
Overwrite Mnemo files Templates or generated files must be refreshed intentionally. npx @dinasor/mnemo-cli@latest --force

The current CLI exposes: --enable-vector, --vector-provider, --dry-run, --force, --project-name, --repo-root, --yes, and --help.

2.1 Environment Preconditions

Mnemo itself requires Node.js 18+. Git is strongly expected because hooks and project context are part of the operating model. Vector mode additionally expects Python 3.10+, pip, and the Python packages needed for the selected provider. In the current installer code path, vector setup installs openai, sqlite-vec, mcp[cli], pyyaml, and google-genai when Gemini is selected.

3. Generated Layout and File Roles

After installation, treat .mnemo/memory/ as canonical. The bridge paths under .cursor/ and .agent/ exist so tools with older assumptions can still load the same memory system. Edit the canonical files unless a specific tool absolutely requires the bridge path.

Always-read layer

hot-rules.md, active-context.md, and memo.md are the highest-value artifacts. They are meant to be small, current, and read in order.

Lessons

lessons/L-XXX-*.md stores reusable prevention knowledge. Each lesson has YAML frontmatter, a stable ID, scoped triggers, and an imperative rule.

Journal and digests

journal/YYYY-MM.md is chronological history. digests/*.digest.md are generated low-token summaries and should be searched before raw journals.

Indexes

lessons/index.md, journal-index.md, and JSON indexes are rebuilt artifacts that support cheap retrieval and inspection.

Artifact Purpose Authoring rule
hot-rules.md Hard invariants and retrieval protocol. Keep under roughly twenty lines. Move detail elsewhere.
active-context.md Session scratchpad with temporary priority. Clear it when the current task is done.
memo.md Long-term current truth: ownership, invariants, runbooks. Store current state, not history.
lessons/ Durable reusable rules derived from incidents or recurrent patterns. One lesson per idea; use canonical tags only.
journal/ Chronological archaeology. Append by date; do not duplicate a date heading within one month file.
digests/ Token-cheap monthly summaries. Generated by rebuild, not hand-maintained.

The template authority order is strict: lessons override everything, active-context.md overrides memo and journal for the current session, memo.md is long-term truth, and the journal is history. If users violate this ordering, retrieval quality declines because agents receive contradictory or stale guidance.

4. Daily Authoring Workflow

The operational sequence after installation is consistent across repositories. First seed the always-read layer with high-signal context. Then rebuild indexes, lint the memory system, and only after that rely on retrieval. Mnemo ships a dedicated skill, mnemo-codebase-optimizer, for rapidly mapping an existing codebase into memory, but the underlying files remain plain Markdown and can be managed manually.

4.1 First-Run Seeding Protocol

  1. Write or refine hot-rules.md with hard invariants and authority order only.
  2. Populate memo.md with ownership, critical flows, commands, and constraints.
  3. Use active-context.md as a temporary scratchpad while seeding.
  4. Create starter lessons for predictable traps, not for generic documentation.
  5. Add an initial journal entry describing what was seeded and what remains uncertain.
  6. Run rebuild and lint before trusting search results.

Mnemo also provides explicit authoring helpers. Lesson creation is formalized because lessons are meant to be atomic and machine-parseable. Journal entries are formalized because duplicate date headings and ad hoc tags destroy retrieval quality over time.

powershell -File scripts/memory/add-lesson.ps1 `
  -Title "Always validate provider env before vector sync" `
  -Tags "Reliability,Process" `
  -Rule "Validate provider configuration and API keys before running vector sync"

powershell -File scripts/memory/add-journal-entry.ps1 `
  -Tags "Process,Docs" `
  -Title "Seeded Mnemo memory for the repository" `
  -Files "README.md,paper/field-guide.html"

The lesson helper assigns the next lesson ID automatically, canonicalizes tags against tag-vocabulary.md, and generates the expected frontmatter. The journal helper appends to the correct month file and merges into an existing day heading when necessary.

5. Retrieval and Search Workflow

The retrieval contract is file-first. Agents should read the always-read layer every time, then search cheap indexes before opening heavy history. The generated index.md template states the protocol directly: always read hot-rules.md, active-context.md, and memo.md; search lesson and journal indexes before opening raw artifacts.

Step What to open Why this order exists
1 hot-rules.md Hard invariants first.
2 active-context.md Session-local overrides second.
3 memo.md Project truth before search fan-out.
4 lessons/index.md Find relevant lesson IDs cheaply.
5 Specific lesson files Open only matched durable guidance.
6 Digests before journals Use compressed history before archaeology.

The PowerShell search helper supports file-based scanning by default and can delegate to SQLite FTS when a local memory.sqlite exists:

powershell -File scripts/memory/query-memory.ps1 -Query "ownership map"
powershell -File scripts/memory/query-memory.ps1 -Query "vector sync" -Area Memo
powershell -File scripts/memory/query-memory.ps1 -Query "known pitfalls" -Format AI
powershell -File scripts/memory/query-memory.ps1 -Query "journal" -UseSqlite

In AI mode the helper prints file references rather than excerpts. That mode is important for agent workflows because it narrows the reading set without flooding context. SQLite mode is optional and only activates when Python, the query helper, and memory.sqlite are available.

5.1 Script Wrapper Note

The public documentation and hook flow reference paired .sh and .ps1 helpers. In the current Node installer implementation, the PowerShell templates are the visible canonical helper sources, while POSIX wrappers are referenced by hooks and the installer summary as generated or existing companions. The usage model is cross-platform, but the repo source of truth for helper logic is currently PowerShell-first.

6. Vector Mode

Vector mode is the semantic layer. When enabled, Mnemo installs scripts/memory/mnemo_vector.py, vector search rules, MCP configuration for Cursor, and the autonomous Python runtime modules. This is the mode to use when lexical search is insufficient, when you need semantic memory recall in the IDE, or when you want no-human-in-the-loop maintenance.

The vector CLI exposes five subcommands in the current codebase: sync, search, forget, health, and status.

Command Purpose Typical use
sync Rebuild vector index from memory markdown files. Run after a first install or significant memory edits.
search <query> --top-k N Semantic search over indexed memory. Use for concept search rather than exact string lookup.
forget <ref_path> Remove vectors for a specific source path. Useful after large content removals or bad ingests.
health Verify DB integrity and provider availability. First diagnostic step when semantic search feels wrong.
status Return JSON status for tracked files, vectors, facts, and lifecycle events. Use for operational monitoring and CI-style checks. 
python scripts/memory/mnemo_vector.py sync
python scripts/memory/mnemo_vector.py search "how do we run tests" --top-k 8
python scripts/memory/mnemo_vector.py health
python scripts/memory/mnemo_vector.py status

The vector engine resolves the provider from MNEMO_PROVIDER or from whichever API key is present. In practice, Gemini is auto-selected when GEMINI_API_KEY exists; otherwise OpenAI is used. The engine stores vectors in a local SQLite database and exposes MCP tools such as vector_search, vector_sync, vector_forget, vector_health, and memory_status.

7. Autonomous Runtime

Autonomous mode is activated by vector installation because it depends on the vector stack and the Python-based lifecycle modules. The runtime is orchestrated by scripts/memory/autonomy/runner.py and executes a full detect-ingest-resolve-lifecycle-promote loop.

Component Role in the loop
ingest_pipeline.py Detects changes and chunks changed memory files into memory units.
entity_resolver.py Extracts and normalizes entities for better retrieval and linking.
lifecycle_engine.py Applies ADD, UPDATE, DEPRECATE, or NOOP decisions to facts.
context_safety.py Applies duplicate, contradiction, and token-budget checks to retrieval packs.
reranker.py Combines semantic, authority, temporal, and entity signals.
policies.yaml Controls thresholds, budgets, and contradiction behavior. 
python scripts/memory/autonomy/runner.py --mode once
python scripts/memory/autonomy/runner.py --mode schedule
python scripts/memory/autonomy/runner.py --mode once --repo D:\path\to\repo

The runner is designed to respond to Git hooks such as post-commit, post-merge, and post-checkout, and it also supports a scheduler loop. A lock file under .autonomy/runner.lock prevents overlapping cycles. If the runtime ingests new knowledge, it writes an autonomous journal delta and may promote stable signals into new lessons.

8. IDE Playbooks

The generated assets differ slightly by agent environment, but the logic is shared: Mnemo keeps one canonical memory store and projects the right rule files or bridge paths for each tool.

Environment What to verify after install Normal operator action
Cursor .mnemo/mcp/cursor.mcp.json, bridge to .cursor/mcp.json, rule files in .mnemo/rules/cursor/. Restart Cursor, run vector_health, then vector_sync if vector mode is enabled.
Claude Code .mnemo/memory/ plus visible .cursor/memory/ bridge. Start from repo root and follow the retrieval order from index.md.
OpenAI Codex Canonical memory store exists and repo is opened from the correct root. Use the same always-read and search-first protocol as Cursor and Claude Code.
Gemini Antigravity .agent/rules/ contains the generated memory-system rule files. Ensure Antigravity actually loads project rule files from that directory.
Other IDEs Point the tool at .mnemo/memory/ or the bridge path. Adopt Mnemo's read order and maintenance commands even if MCP is not used.

Mnemo also installs a repo-local skill under .cursor/skills/mnemo-codebase-optimizer/. That skill is the fastest way to seed high-signal memory for an existing codebase because it explicitly maps architecture, ownership, workflows, and starter lessons before running rebuild, lint, and optional sync.

9. Maintenance and Recovery

Mnemo remains useful only if memory stays curated. The basic maintenance loop is simple: rebuild indexes after edits, lint for structural errors, clear active-context.md when the task is done, and promote repeated journal knowledge into lessons rather than letting history bloat the always-read layer.

powershell -File scripts/memory/rebuild-memory-index.ps1
powershell -File scripts/memory/lint-memory.ps1
powershell -File scripts/memory/clear-active.ps1

The rebuild helper regenerates lesson and journal indexes, monthly digests, and optionally the SQLite search database. The linter checks lesson frontmatter, duplicate IDs, allowed tags, journal date hygiene, and the approximate size of the always-read layer. The runtime path now uses token-aware counting for context pack safety, but the linter still acts as a rough chars-to-tokens guardrail for authoring discipline.

Symptom Likely cause Operator response
Search returns noisy or stale files Indexes were not rebuilt after memory edits. Run rebuild, then rerun the query.
Lint fails on lessons Missing YAML fields, unknown tags, or duplicate lesson IDs. Repair the lesson frontmatter and rerun lint.
Semantic search fails Provider key missing, bad dependency install, or unhealthy vector DB. Run vector_health and inspect provider configuration.
Autonomous runner never starts Lock file is stale or vector mode was never enabled. Check .autonomy/runner.lock and verify vector installation.
Memory becomes bloated Historical detail was stored in memo or active context instead of journal and lessons. Move history to journal, summarize patterns into lessons, and trim memo.

10. Verification Checklist

Mnemo ships with meaningful checks. Use them. A valid installation is not just one that wrote files; it is one that can rebuild, lint, retrieve, and, if enabled, search semantically without operator surprise.

sh ./tests/test-installer.sh
python3 -m py_compile scripts/memory/installer/templates/autonomy/*.py
python3 tests/retrieval/benchmark_runner.py --fixtures tests/retrieval/fixtures/
python3 tests/retrieval/benchmark_runner.py --fixtures tests/retrieval/fixtures/ --ablation
python3 tests/retrieval/eval_contradiction.py --by-category
python3 tests/retrieval/weight_sweep.py --curve-json tests/retrieval/reports/weight_curve.json

For day-to-day repo use, the short verification loop is even simpler: install, seed core memory, rebuild, lint, test retrieval with a handful of real questions, and only then rely on the memory layer in agent sessions. If vector mode exists, add vector_sync and vector_health to that loop.

12. Conclusion

Mnemo works best when treated as an operating discipline. The installer gives structure, but the payoff comes from maintaining a small always-read layer, promoting durable lessons, keeping journals chronological, rebuilding indexes routinely, and enabling vector or autonomous mode only when the repository is ready to support them. Used this way, Mnemo becomes a stable shared memory surface across IDEs and agent runtimes rather than another stale documentation folder.
