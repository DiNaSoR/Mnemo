Mnemo: A Token-Budget-Aware Structured Memory System for AI Coding Agents

DiNaSoR

Open-Source Software · GitHub: DiNaSoR/Mnemo

March 2026 · Version 0.0.8 · Companion guide: Operational Field Guide

Abstract

AI coding agents are effective but stateless: they repeatedly rediscover repository conventions and can reintroduce previously resolved mistakes. Mnemo is a repository-native memory layer that structures project memory into typed artifacts and supports retrieval through optional vector search and deterministic scoring. This version focuses on measurable behavior. We replace contradiction checks based on token overlap with predicate-aware fact-frame matching (plus heuristic fallback), migrate context budgets to token-based accounting, expand evaluation fixtures from 7 to 32 categorized queries, and add ablation and contradiction-quality evaluation scripts. On the current benchmark suite (FTS mode), Mnemo reports hit@3 = 0.875, nDCG@5 = 0.7824, and MRR = 0.8578. Contradiction evaluation on 64 labeled pairs reports precision = 1.00, recall = 0.7812, and F1 = 0.8772. We position Mnemo as a practical systems/tool contribution rather than a new foundation model or state-of-the-art memory architecture.

Keywords: AI coding agents, structured memory, token budgeting, retrieval evaluation, contradiction detection, benchmark ablation, IDE interoperability

1. Introduction

Project memory for coding agents is commonly implemented through ad-hoc files and IDE-specific conventions. These mechanisms can be useful but often lack lifecycle governance, contradiction handling, and transparent evaluation. Mnemo addresses this by scaffolding a canonical in-repo memory layer and bridging it to IDE-native mechanisms such as Cursor rules and instruction manifests.

This revision narrows claims to measured outcomes and contributes:

  1. Token-aware context safety: token budgets are enforced with tokenizer-backed counting, with explicit fallback mode reporting.
  2. Improved contradiction detection: predicate-aware fact-frame matching with heuristic and optional embedding enhancement.
  3. Evaluation upgrades: 32 categorized retrieval fixtures, ablation mode, contradiction precision/recall/F1, and weight sensitivity search.

2. Related Work

Mnemo sits at the intersection of agent memory, RAG ranking, and tool interoperability.

  • Agent memory architectures: MemGPT [1], Generative Agents [2], Reflexion [3], and A-MEM [4] explore long-horizon memory behavior and reflection loops at the agent-policy level.
  • Production memory tooling: Mem0 [5] and similar systems focus on practical memory APIs and persistence for LLM applications.
  • Repository instruction mechanisms: Cursor rules, CLAUDE.md, AGENTS.md, and Continue rules files [6][7][8] provide local control surfaces but usually without autonomous lifecycle governance.
  • Retrieval foundations: RAG [9], BM25-style relevance [10], and vector indexing backends such as sqlite-vec [15] inform Mnemo's retrieval stack.

Compared with these systems, Mnemo primarily contributes repository-native operationalization: memory structure, ranking heuristics, lifecycle automation, and testable evaluation scripts in one OSS CLI package.

3. System Design

Mnemo has three runtime layers:

  1. Memory representation: typed markdown artifacts (core, procedural, episodic, semantic/resource) plus optional vector indexing.
  2. Autonomous lifecycle: ingest, entity resolution, fact lifecycle, contradiction handling, and lesson promotion.
  3. Retrieval and safety: score-fusion reranking (semantic + authority + temporal + entity) and context safety filtering.

Key algorithm updates in this revision:

  • Contradiction detection: replaced Jaccard-only checks with predicate-aware fact-frame matching, hybrid fallback heuristics, and optional local embedding confirmation.
  • Token accounting: context budgets now measured in tokens with provider-aware counters, not character approximations.
  • Entity extraction: expanded to repeated snake_case identifiers, hyphenated packages, abbreviations, and policy-defined custom entities.
  • Weight configurability: score-fusion weights are now constructor-configurable and benchmark-tunable.

4. Evaluation

4.1 Setup

We evaluate with fixture-only offline retrieval tests and contradiction pair evaluation scripts included in the repository. Retrieval fixtures now include 32 categorized queries across basic, procedural, episodic, and entity groups.

Category-level results show strongest performance on entity-focused queries and lower performance on episodic and procedural subsets, which are targets for further recall improvements.

4.3 Contradiction Detection

On 64 labeled sentence pairs (16 per category):

  • Legacy Jaccard baseline: precision = 0.00, recall = 0.00, F1 = 0.00.
  • Current detector (hybrid mode): precision = 1.00, recall = 0.7812, F1 = 0.8772.

Category analysis shows high recall on direct negation (0.9375) and lower recall on modal conflicts (0.625), motivating future NLI-backed contradiction handling.

4.4 Weight Sensitivity

Weight sweep (step 0.25) indicates the best composite config in this benchmark slice: semantic=0.50, authority=0.00, temporal=0.25, entity=0.25 (score 0.8464). The default profile remains competitive (score 0.8235), and the generated distance-to-default sensitivity curve shows quality degradation as weights move farther from practical operating ranges.

5. Discussion

5.1 Scope and Positioning

Mnemo is a systems/tooling contribution. It is not a new general-memory architecture for LLMs and is not presented as state-of-the-art on external long-context benchmarks. The value proposition is reproducible, repository-native memory operations with practical quality instrumentation.

5.2 Limitations

  • Deterministic retrieval ordering does not guarantee deterministic model behavior.
  • Predicate/heuristic contradiction logic can still miss implicit contradictions and nuanced modality shifts.
  • Entity extraction remains pattern-based and may under-recall domain-specific conceptual entities without custom policy entries.
  • Vector-mode quality depends on provider/model choices and environment configuration.
  • sqlite-vec is pre-v1 and may introduce compatibility churn over time.

5.3 Future Work

  • Optional NLI-backed contradiction scoring.
  • RRF or learned ranking alternatives to linear score fusion.
  • External benchmark evaluation and larger cross-repository fixture sets.
  • Additional security hardening for prompt-injection and integrity protections.

6. Conclusion

Mnemo provides a practical memory layer for AI coding agents with token-budget-aware context filtering, contradiction-aware lifecycle management, and measurable retrieval quality instrumentation. This revision improves empirical grounding by expanding fixtures, adding ablations, and reporting contradiction and weight sensitivity results in-repo.
